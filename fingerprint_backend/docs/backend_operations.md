## ملاحظات عامة حول الاتصال والمصادقة

* **عنوان الخادم الأساسي (Base URL):** `http://localhost:8000` (أو المنفذ الذي يعمل عليه الخادم).
* **بروتوكول المصادقة:** تستخدم العمليات التي تتطلب مصادقة ترويسة الطلب `Authorization` من نوع `Bearer Token`:
  ```http
  Authorization: Bearer <Access_Token>
  ```
* **أنواع البيانات المستخدمة:**
  - `string`: نصوص.
  - `int`: أرقام صحيحة.
  - `float`: أرقام عشرية.
  - `bool`: قيم منطقية (`true` / `false`).
  - `DateTime`: نصوص تمثل الوقت والتاريخ بالتنسيق القياسي ISO 8601 (مثال: `2026-07-12T03:13:42`).
  - `date`: نصوص تمثل التاريخ بالتنسيق `YYYY-MM-DD` (مثال: `2026-07-12`).
  - `nullable`: حقول اختيارية يمكن أن تكون قيمتها فارغة (`null`).

## 1. التوثيق والمصادقة (Auth)

تتعامل هذه الوحدة مع تسجيل دخول المستخدمين وإصدار توكنات الوصول JWT وتحديثها.

### 1.1 تسجيل الدخول - Login
* **المسار وطريقة الطلب:** `POST /api/auth/login`
* **المصادقة:** لا يتطلب.
* **المدخلات (JSON Body):**
  | اسم الحقل | نوع البيانات | حالة الحقل | الوصف |
  | :--- | :--- | :--- | :--- |
  | `username` | `string` | مطلوب | اسم المستخدم في النظام |
  | `password` | `string` | مطلوب | كلمة المرور |

  *مثال للطلب:*
  ```json
  {
    "username": "admin",
    "password": "mysecretpassword"
  }
  ```

* **المخرجات (Response Body):**
  * **حالة النجاح (200 OK):**
    | اسم الحقل | نوع البيانات | الوصف |
    | :--- | :--- | :--- |
    | `token` | `string` | توكن الوصول (Access Token) الصالح للمصادقة |
    | `refreshToken` | `string` | توكن التحديث (Refresh Token) المستخدم لتوليد توكن جديد |
    | `user` | `object` | كائن يحتوي على بيانات المستخدم (UserResponse) |
    | `user.id` | `int` | المعرف الفريد للمستخدم |
    | `user.username`| `string` | اسم المستخدم |
    | `user.role` | `string` | دور المستخدم في النظام (`admin`, `hr`, `viewer` إلخ) |
    | `user.employee_id`| `int` أو `null` | معرف الموظف المرتبط بهذا المستخدم (إن وُجد) |

    *مثال للاستجابة الناجحة:*
    ```json
    {
      "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
      "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
      "user": {
        "id": 1,
        "username": "admin",
        "role": "admin",
        "employee_id": null
      }
    }
    ```
  * **الأخطاء المحتملة:**
    - `401 Unauthorized`: عند إدخال بيانات اعتماد غير صحيحة (`{"detail": "Invalid credentials"}`).
    - `422 Unprocessable Entity`: عند نقص أحد الحقول المطلوبة.

---

### 1.2 تحديث التوكن - Refresh Token
* **المسار وطريقة الطلب:** `POST /api/auth/refresh`
* **المصادقة:** لا يتطلب (يمرر التوكن في الجسم).
* **المدخلات (JSON Body):**
  | اسم الحقل | نوع البيانات | حالة الحقل | الوصف |
  | :--- | :--- | :--- | :--- |
  | `refreshToken` | `string` | مطلوب | رمز التحديث المحفوظ سابقاً لدى العميل |

  *مثال للطلب:*
  ```json
  {
    "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
  ```

* **المخرجات (Response Body):**
  * **حالة النجاح (200 OK):**
    | اسم الحقل | نوع البيانات | الوصف |
    | :--- | :--- | :--- |
    | `token` | `string` | توكن الوصول الجديد |
    | `refreshToken` | `string` | توكن التحديث الجديد |

    *مثال للاستجابة الناجحة:*
    ```json
    {
      "token": "eyJhbGciOiJIUzI1Ni...",
      "refreshToken": "eyJhbGciOiJIUzI1Ni..."
    }
    ```
  * **الأخطاء المحتملة:**
    - `401 Unauthorized`: في حال كان التوكن منتهي الصلاحية أو غير صالح، أو إذا كان المستخدم غير موجود أو غير نشط.

---

### 1.3 تسجيل الخروج - Logout
* **المسار وطريقة الطلب:** `POST /api/auth/logout`
* **المصادقة:** يتطلب (`Bearer Token`).

  * **حالة النجاح (200 OK):**
    ```json

      "message": "Logged out successfully"
    }
    ```

        "id": 1,

## 2. إدارة الموظفين (Employees)

توفر هذه الوحدة جميع عمليات CRUD (إنشاء، قراءة، تعديل، حذف) لبيانات الموظفين في قاعدة البيانات.

### 2.1 جلب قائمة الموظفين النشطين - Get Active Employees
* **المسار وطريقة الطلب:** `GET /api/employees/`
* **المصادقة:** يتطلب (`Bearer Token`).
* **المدخلات:** لا يوجد.
* **المخرجات (Response Body):**
  * **حالة النجاح (200 OK):** قائمة من كائنات الموظفين `list[EmployeeResponse]`:
    | اسم الحقل | نوع البيانات | الوصف |
    | :--- | :--- | :--- |
    | `uid` | `int` | المعرف التسلسلي الفريد للموظف في قاعدة البيانات |
    | `employee_id` | `string` | المعرف النصي المخصص للموظف (كود الموظف، مثلاً `EMP001`) |
    | `name` | `string` | الاسم الكامل للموظف |
    | `role` | `string` | دور الموظف (الافتراضي: `"user"`) |
    | `group_id` | `string` أو `null` | معرف المجموعة المرتبط بها الموظف |
    | `card_no` | `int` أو `null` | رقم بطاقة الموظف البيومترية إن وجد |
    | `default_shift_id`| `int` أو `null` | معرف الوردية الافتراضية المرتبطة بالموظف |
    | `is_active` | `bool` | حالة نشاط الموظف (نشط `true` / غير نشط `false`) |

    *مثال للاستجابة الناجحة:*
    ```json
    [
      {
        "uid": 1,
        "employee_id": "EMP001",
        "name": "أحمد محمد",
        "role": "user",
        "group_id": "GRP-A",
        "card_no": 123456,
        "default_shift_id": 2,
        "is_active": true
      }
    ]
    ```

---

### 2.2 إنشاء موظف جديد - Create Employee
* **المسار وطريقة الطلب:** `POST /api/employees/`
* **المصادقة:** يتطلب (`Bearer Token`).
* **المدخلات (JSON Body):** كائن من نوع `EmployeeCreate`:
  | اسم الحقل | نوع البيانات | حالة الحقل | الوصف |
  | :--- | :--- | :--- | :--- |
  | `employee_id` | `string` | مطلوب | الكود النصي للموظف (يجب أن يكون فريداً) |
  | `name` | `string` | مطلوب | اسم الموظف |
  | `role` | `string` | اختياري | دور الموظف (القيمة الافتراضية: `"user"`) |
  | `group_id` | `string` | اختياري | معرف المجموعة |
  | `card_no` | `int` | اختياري | رقم بطاقة الموظف |
  | `default_shift_id`| `int` | اختياري | معرف الوردية الافتراضية للموظف |
  | `password` | `string` | اختياري | كلمة المرور الخاصة بالموظف (يتم تشفيرها تلقائياً بالخلفية) |

  *مثال للطلب:*
  ```json
  {
    "employee_id": "EMP002",
    "name": "خالد العتيبي",
    "role": "user",
    "group_id": "GRP-B",
    "card_no": 789012,
    "default_shift_id": 1,
    "password": "mypassword123"
  }
  ```

* **المخرجات (Response Body):**
  * **حالة النجاح (201 Created):** يعود بكائن الموظف المنشأ حديثاً من نوع `EmployeeResponse` مع توليد الـ `uid`.

    *مثال للاستجابة الناجحة:*
    ```json
    {
      "uid": 2,
      "employee_id": "EMP002",
      "name": "خالد العتيبي",
      "role": "user",
      "group_id": "GRP-B",
      "card_no": 789012,
      "default_shift_id": 1,
      "is_active": true
    }
    ```
  * **الأخطاء المحتملة:**
    - `409 Conflict`: إذا كان كود الموظف `employee_id` مسجلاً مسبقاً في النظام.

---

### 2.3 جلب بيانات موظف محدد - Get Employee By UID
* **المسار وطريقة الطلب:** `GET /api/employees/{employee_uid}`
* **المصادقة:** يتطلب (`Bearer Token`).
* **المدخلات:**
  - **محدد مسار (Path Parameter):** `employee_uid` (نوع: `int`) - المعرف الرقمي الفريد للموظف.
* **المخرجات (Response Body):**
  * **حالة النجاح (200 OK):** كائن الموظف المطلوب (`EmployeeResponse`).
  * **الأخطاء المحتملة:**
    - `404 Not Found`: إذا كان الرقم التعريفي للموظف غير موجود في النظام.

---

### 2.4 تحديث بيانات موظف - Update Employee
* **المسار وطريقة الطلب:** `PUT /api/employees/{employee_uid}`
* **المصادقة:** يتطلب (`Bearer Token`).
* **المدخلات:**
  - **محدد مسار (Path Parameter):** `employee_uid` (نوع: `int`).
  - **جسم الطلب (JSON Body):** كائن من نوع `EmployeeUpdate` (جميع حقوله اختيارية، يرسل فقط ما يراد تعديله):
    | اسم الحقل | نوع البيانات | الوصف |
    | :--- | :--- | :--- |
    | `employee_id` | `string` | الكود النصي للموظف (يجب أن يبقى فريداً) |
    | `name` | `string` | اسم الموظف |
    | `role` | `string` | دور الموظف |
    | `group_id` | `string` | معرف المجموعة |
    | `card_no` | `int` | رقم البطاقة |
    | `default_shift_id`| `int` | معرف الوردية الافتراضية |
    | `password` | `string` | كلمة مرور جديدة (يتم تشفيرها تلقائياً) |
    | `is_active` | `bool` | حالة الموظف (نشط أو معطل) |

* **المخرجات (Response Body):**
  * **حالة النجاح (200 OK):** كائن الموظف بعد التعديل (`EmployeeResponse`).
  * **الأخطاء المحتملة:**
    - `404 Not Found`: إذا لم يتم العثور على الموظف.
    - `409 Conflict`: إذا تم تعديل `employee_id` إلى قيمة مستخدمة بالفعل لموظف آخر.

---

### 2.5 حذف موظف (تعطيل النشاط) - Deactivate Employee
* **المسار وطريقة الطلب:** `DELETE /api/employees/{employee_uid}`
* **المصادقة:** يتطلب (`Bearer Token`).
* **المدخلات:**
  - **محدد مسار (Path Parameter):** `employee_uid` (نوع: `int`).
* **المخرجات (Response Body):**
  * **حالة النجاح (200 OK):** رسالة نجاح التعطيل (الـ Backend يستخدم الحذف الناعم Soft Delete بحيث يتم تغيير حقل `is_active` إلى `false` ولا يتم حذفه من قاعدة البيانات نهائياً حفاظاً على سلامة السجلات المرتبطة).
    ```json
    {
      "message": "Employee deactivated successfully"
    }
    ```
  * **الأخطاء المحتملة:**
    - `404 Not Found`: إذا لم يتم العثور على الموظف.

## 3. الحضور والانصراف (Attendance)

توفر هذه الوحدة سجلات البصمات الخام المستلمة من الأجهزة البيومترية، بالإضافة إلى السجلات المعالجة التي تم حساب ساعات العمل والتأخير والعمل الإضافي فيها.

### 3.1 جلب سجلات الحضور الخام - Get Attendance Logs
* **المسار وطريقة الطلب:** `GET /api/attendance/logs`
* **المصادقة:** يتطلب (`Bearer Token`).
* **المدخلات (Query Parameters):**
  | اسم المعلم | نوع البيانات | حالة المعلم | الوصف |
  | :--- | :--- | :--- | :--- |
  | `employee_id` | `int` | اختياري | تصفية السجلات لموظف محدد (باستخدام المعرف الرقمي `uid`) |
  | `device_id` | `int` | اختياري | تصفية السجلات لجهاز بيومتري محدد |
  | `limit` | `int` | اختياري | الحد الأقصى للنتائج المسترجعة (الافتراضي: 100، المدى: 1 إلى 1000) |
  | `offset` | `int` | اختياري | الإزاحة لجلب النتائج على دفعات (Pagination) (الافتراضي: 0) |

* **المخرجات (Response Body):**
  * **حالة النجاح (200 OK):** قائمة من كائنات سجلات الحضور الخام `list[AttendanceLogResponse]`:
    | اسم الحقل | نوع البيانات | الوصف |
    | :--- | :--- | :--- |
    | `id` | `int` | المعرف الفريد لسجل البصمة |
    | `employee_id` | `int` أو `null` | معرف الموظف المرتبط بالبصمة (يكون `null` إذا كانت البصمة غير معرفة) |
    | `unrecognized_biometric` | `string` أو `null` | نص البصمة غير المعرفة في حال عدم مطابقتها لأي موظف |
    | `device_id` | `int` أو `null` | معرف الجهاز البيومتري الذي تمت البصمة عليه |
    | `punch_time` | `DateTime` | تاريخ ووقت البصمة |

    *مثال للاستجابة الناجحة:*
    ```json
    [
      {
        "id": 15,
        "employee_id": 1,
        "unrecognized_biometric": null,
        "device_id": 2,
        "punch_time": "2026-07-12T08:05:00"
      }
    ]
    ```

---

### 3.2 جلب سجلات الحضور المعالجة - Get Processed Attendance Records
* **المسار وطريقة الطلب:** `GET /api/attendance/records`

* **المدخلات (Query Parameters):**
  | اسم المعلم | نوع البيانات | حالة المعلم | الوصف |
  | :--- | :--- | :--- | :--- |
  | `employee_id` | `int` | اختياري | تصفية النتائج لموظف محدد |

* **المخرجات (Response Body):**
  * **حالة النجاح (200 OK):** قائمة من كائنات السجلات المعالجة يومياً `list[AttendanceRecordResponse]`:
    | اسم الحقل | نوع البيانات | الوصف |
    | :--- | :--- | :--- |
    | `id` | `int` | معرف السجل المعالج |
    | `employee_id` | `int` | معرف الموظف |
    | `payroll_period_id`| `int` | معرف فترة الراتب المعتمدة المرتبط بها اليوم |

    | `lateness_mins` | `int` | عدد دقائق التأخر عن بداية الوردية المعتمدة |
    | `overtime_mins` | `int` | عدد دقائق العمل الإضافي المحسوبة والمعتمدة |
    | `is_locked` | `bool` | حالة قفل السجل (تصبح `true` عند إغلاق فترة الرواتب لمنع التعديل) |
    | `flags` | `string` أو `null` | ملاحظات أو رايات خاصة بالنظام حول اليوم (مثل: `"ABSENT"` للغياب) |

    *مثال للاستجابة الناجحة:*
    ```json
    [
      {
        "id": 102,
        "employee_id": 1,
        "payroll_period_id": 3,
        "record_date": "2026-07-11",
        "total_hours": 8.0,
        "lateness_mins": 15,
        "overtime_mins": 60,
        "is_locked": false,
        "flags": null
      }
    ]
    ```

## 4. إدارة الورديات (Shifts)

تتحكم هذه الوحدة بالورديات المتوفرة بالمؤسسة وتوقيتاتها وفترات السماح.

### 4.1 جلب قائمة الورديات - Get Shifts
* **المسار وطريقة الطلب:** `GET /api/shifts/`
* **المصادقة:** يتطلب (`Bearer Token`).
* **المدخلات:** لا يوجد.
* **المخرجات (Response Body):**
  * **حالة النجاح (200 OK):** قائمة من كائنات الورديات `list[ShiftResponse]`:
    | اسم الحقل | نوع البيانات | الوصف |

## 11. اتصال الأجهزة (WebSocket)

يستخدم لإنشاء اتصال مستمر ثنائي الاتجاه (Bi-directional) ومنخفض التأخير بين خادم الـ Backend وأجهزة البصمة البيومترية المثبتة.

### 11.1 مسار الاتصال - WebSocket Endpoint
* **المسار وطريقة الطلب:** `WS /ws/devices`
* **المصادقة:** لا تتطلب مصادقة JWT مباشرة عبر الترويسة، بل يتم التحقق من رقم معرّف الجهاز المسجل فور نجاح الاتصال الأولي من خلال رسالة تسجيل مخصصة.

### 11.2 بروتوكول الرسائل (Message Format)
يتم نقل الرسائل على هيئة نصوص منسقة بصيغة JSON.

#### أ) رسالة تسجيل اتصال الجهاز (تُرسل من الجهاز فور الاتصال)
* **نوع الرسالة (Direction):** من الجهاز ← إلى الخادم
* **بنية الرسالة:**
  ```json
  {
    "type": "register",
    "device_id": 1
  }
  ```
* **رد الخادم للـ الجهاز (Server Response):**
  * *في حال النجاح:*
    ```json
    {
      "type": "registered",
      "device_id": 1,
      "message": "Device registered successfully"
    }
    ```
  * *في حال فشل العثور على الجهاز بقاعدة البيانات:*
    ```json
    {
      "type": "error",
      "message": "Device not found"
    }
    ```

#### ب) رسالة نبضات القلب - Heartbeat (تُرسل دورياً للتأكد من حيوية الجهاز)
* **نوع الرسالة (Direction):** من الجهاز ← إلى الخادم
* **بنية الرسالة:**
  ```json
  {
    "type": "heartbeat",
    "device_id": 1
  }
  ```
* **رد الخادم للـ الجهاز (Server Response):**
  ```json
  {
    "type": "heartbeat_ack",
    "device_id": 1,
    "timestamp": "2026-07-12T03:15:00"
  }
  ```
  *(الخادم يقوم تلقائياً بتحديث حقل `is_online` إلى `true` وحقل `last_sync` بـ وقت الاستلام).*

#### ج) رسالة تسجيل بصمة الموظف الحية - Biometric Data (عندما يبصم موظف)
* **نوع الرسالة (Direction):** من الجهاز ← إلى الخادم
* **بنية الرسالة:**
  ```json
  {
    "type": "biometric_data",
    "device_id": 1,
    "biometric_id": "FP001",
    "is_check_in": true,
    "timestamp": "2026-07-12T08:05:00"
  }
  ```
* **رد الخادم للـ الجهاز (Server Response):**
  ```json
  {
    "type": "biometric_ack",
    "log_id": 124,
    "device_id": 1,
    "message": "Biometric data recorded"
  }
  ```

#### د) رسالة طلب حالة الجهاز الحالية
* **نوع الرسالة (Direction):** من الجهاز ← إلى الخادم
* **بنية الرسالة:**
  ```json
  {
    "type": "status",
    "device_id": 1
  }
  ```
* **رد الخادم للـ الجهاز (Server Response):**
  ```json
  {
    "type": "status_response",
    "device_id": 1,
    "is_online": true,
    "last_sync": "2026-07-12T03:15:00"
  }
  ```
